````md
## Fabric 课件编辑器数据设计与流转总结

### 1. 核心设计思想

这个编辑器里，我没有把 Fabric 对象作为业务数据源，而是把 store 里的课件数据作为唯一可信数据源。

```txt
Store = 业务数据模型
Fabric = 渲染层 + 交互层
React UI = 操作入口 + 状态展示
```
````

也就是说：

```txt
数据最终落在 Zustand store
Fabric 对象只是 store 数据在画布上的运行时投影
```

这样更方便做：

```txt
撤销 / 重做
保存 / 导出
图层管理
多页面
属性面板
性能优化
后续协同编辑
```

---

## 2. Store 数据结构

核心数据大概分成两类：文档数据和 UI 状态。

### 文档数据

```ts
pages: Record<string, PageData>;
widgets: Record<string, Widget>;
rootIds: Record<string, string[]>;
childIds: Record<string, string[]>;
```

含义：

```txt
pages
  页面信息，比如宽、高、背景色

widgets
  所有元素的扁平数据池，key 是 widgetId

rootIds
  每个页面的根图层顺序，决定画布渲染层级

childIds
  group 节点的子元素关系
```

这里没有直接嵌套树，而是用了扁平结构：

```txt
widgets 存实体
rootIds / childIds 存关系
```

这样做的好处是：

```txt
查找快
更新单个元素方便
层级顺序清晰
删除 / 恢复 / 重排更容易做
```

### UI 状态

```ts
activePageId;
selectedIds;
hoveredId;
editingTextId;
zoom;
pan;
```

这些是编辑器运行状态，不一定属于课件内容本身。

---

## 3. Widget 和 Fabric 的关系

每个 `Widget` 是可序列化的数据模型：

```ts
{
  (id,
    type,
    name,
    pageId,
    parentId,
    left,
    top,
    width,
    height,
    angle,
    scaleX,
    scaleY,
    fill,
    stroke,
    opacity,
    visible,
    locked);
}
```

Fabric 对象通过 `widgetToFabric(widget)` 创建：

```txt
Widget -> fabric.Object
```

同时会把 widgetId 挂到 Fabric 对象上：

```ts
obj.set({ data: { id: widget.id } });
```

这样 Fabric 发生交互时，可以反查到 store 里的 widget。

---

## 4. 初始化渲染流程

页面加载时：

```txt
PRESET_WIDGETS / PRESET_ROOT_IDS
  -> Zustand store 初始化
  -> Canvas 读取 activePageId
  -> 根据 rootIds 找到当前页 widget
  -> widgetToFabric 创建 fabric.Object
  -> canvas.add
  -> canvasEngine.registerObject(id, obj)
```

数据流：

```txt
store.rootIds + store.widgets
  -> widgetToFabric
  -> fabric.Object
  -> canvas.add
```

---

## 5. React UI 更新 Store

比如右侧属性面板修改颜色：

```txt
用户修改 fill
  -> updateWidget(id, { fill })
  -> history.dispatch(UpdateWidgetCommand)
  -> store._updateWidget
  -> widgets[id] 更新
  -> widgetPatches 记录增量变化
  -> widgetPatchVersion + 1
```

然后 Canvas 同步：

```txt
Canvas 订阅 widgetPatchVersion
  -> 读取 widgetPatches
  -> 找到对应 fabric.Object
  -> applyWidgetPatch
  -> canvas.requestRenderAll
```

也就是说：

```txt
React UI -> Store -> Fabric
```

---

## 6. Fabric 交互回写 Store

比如用户拖动画布上的元素：

```txt
用户拖动 Fabric 对象
  -> Fabric 自己实时移动和渲染
  -> 拖动结束触发 object:modified
  -> 读取 obj.left / top / angle / scale
  -> updateWidget(id, patch)
  -> store 更新
  -> 右侧属性面板刷新
```

数据流：

```txt
Fabric object:modified
  -> updateWidget
  -> Store
  -> RightPanel
```

注意：拖动过程中没有实时写 store，而是拖动结束后写一次。

这样可以避免：

```txt
mousemove 高频触发 store 更新
React 面板高频重渲染
Canvas 重复 patch
```

---

## 7. 为什么不会死循环

Fabric 拖动结束后会写 store，store 又会通知 Canvas patch。

但不会死循环，因为：

```txt
object:modified 是用户交互完成时触发
obj.set / setCoords 是程序式更新
程序式 set 不会再次触发 object:modified
```

所以不会出现：

```txt
Fabric modified
  -> Store update
  -> Fabric set
  -> modified
  -> Store update
```

---

## 8. CanvasEngine 的作用

`CanvasEngine` 持有运行时 Fabric 实例和映射关系：

```ts
widgetId -> fabric.Object
```

它负责：

```txt
select
clearSelection
getObject
registerObject
unregisterObject
bringToFront
sendToBack
align
exportPNG
```

这样上层模块不需要直接操作 Fabric 细节。

---

## 9. History / Command 数据流

文档变更不会直接调用内部 mutator，而是走 history command：

```txt
addWidget
updateWidget
removeWidget
reorderWidget
  -> history.dispatch(Command)
  -> command.execute()
  -> store._xxx()
```

例如更新属性：

```txt
updateWidget
  -> UpdateWidgetCommand
  -> 记录旧值 prev
  -> _updateWidget(patch)
```

撤销时：

```txt
undo
  -> command.undo()
  -> 用 prev 恢复
```

删除时会保存快照：

```txt
RemoveSnapshot {
  pageId,
  parentId,
  index,
  widgets,
  childIds
}
```

撤销删除时：

```txt
_restoreWidgets(snapshot)
```

可以恢复：

```txt
元素本身
子节点
分组关系
原始层级位置
```

---

## 10. 性能优化后的数据流

### 以前的问题

以前一个 widget 更新时，Canvas 会扫当前页所有对象：

```txt
widgets 变化
  -> 遍历 rootIds
  -> 检查每个 widget
  -> 检查层级
  -> patch / reorder
```

500 个对象时，只改一个对象也会扫 500 个。

### 现在的优化

#### 结构变化

```txt
rootIds 变化
  -> 新增 / 删除 / 层级同步
```

#### 属性变化

```txt
widgetPatches 变化
  -> 只 patch 变化过的 widget
```

属性同步从：

```txt
O(n)
```

接近变成：

```txt
O(k)
```

其中：

```txt
n = 当前页对象总数
k = 本次变化对象数
```

---

## 11. 右侧属性面板数据流优化

之前右侧输入框每次 onChange 都会写 store：

```txt
onChange
  -> updateWidget
  -> history
  -> store
  -> canvas patch
  -> render
```

现在改成：

```txt
输入过程
  -> 本地 draft state

提交时
  -> updateWidget
```

比如：

```txt
InputNumber: blur / Enter 提交
Slider: onChangeComplete 提交
ColorPicker: onChangeComplete 提交
```

这样避免右侧表单输入时高频触发全局数据流。

---

## 12. 面试总结话术

这个编辑器的数据模型是以 store 为中心设计的。

Fabric 不是数据源，而是渲染和交互层。课件的页面、图层、元素属性、层级关系都保存在 Zustand store 里。Fabric 对象由 store 里的 widget 数据生成，并通过 `data.id` 和 widgetId 建立映射。

用户通过右侧属性面板或图层面板修改元素时，数据流是：

```txt
React UI -> store -> Canvas patch -> Fabric
```

用户直接拖动画布对象时，数据流是：

```txt
Fabric object:modified -> store -> React UI
```

为了保证撤销重做，所有文档变更都通过 Command / History 管线执行，比如新增、删除、更新、排序都会记录 undo 信息。

为了支持大数据量，我把 Canvas 同步拆成结构同步和属性同步。结构同步只处理 rootIds 变化，属性同步通过 widgetPatches 做增量 patch，避免一个对象变化时扫描整页对象。同时图层面板做虚拟列表和行级订阅，右侧属性面板用本地 draft，提交时才更新 store。

整体目标是：

```txt
store 保证数据可信
Fabric 保证交互体验
History 保证可撤销
增量同步保证大数据量下不卡顿
```
