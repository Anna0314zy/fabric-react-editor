## Fabric 编辑器性能优化总结

### 核心思路

性能优化的核心不是一上来做复杂缓存，而是先拆分数据变化类型，减少无效更新。

```txt
结构变化、属性变化、UI 输入、图层列表渲染
分别走不同路径
```

---

## 1. Canvas 同步拆分

### 问题

之前 `widgets` 或 `rootIds` 一变化，就统一做整页 diff：

```txt
检查新增
检查删除
检查属性变化
检查层级
requestRenderAll
```

这导致只改一个对象颜色，也会触发层级检查。

### 优化

拆成两条路径：

```txt
rootIds 变化
  -> 新增 / 删除 / 层级排序

widget 属性变化
  -> 只 patch 对应 Fabric 对象
```

### 收益

```txt
普通属性更新不再触发层级逻辑
减少无效计算
降低 500+ 对象场景下的同步成本
```

---

## 2. 批量 Canvas 操作只渲染一次

### 问题

初始化 500 个对象时，如果每次 `canvas.add` 都触发渲染，会很重。

### 优化

批量操作期间关闭自动渲染：

```ts
const prev = canvas.renderOnAddRemove;
canvas.renderOnAddRemove = false;

try {
  // batch add / remove / moveObjectTo
} finally {
  canvas.renderOnAddRemove = prev;
}

canvas.requestRenderAll();
```

### 收益

```txt
大量新增 / 删除 / 重排时，只渲染一次
减少初始化和批量操作的重绘成本
```

---

## 3. widgetPatches + widgetPatchVersion 增量更新

### 问题

之前只改一个对象，也要遍历当前页所有对象找变化：

```txt
500 个对象，改 1 个 -> 扫 500 个
```

### 优化

store 中记录本轮变化：

```ts
widgetPatches: {
  [widgetId]: patch
}

widgetPatchVersion: number
```

Canvas 只订阅：

```ts
widgetPatchVersion;
```

然后消费：

```ts
widgetPatches;
```

### 收益

属性同步从：

```txt
O(n) n = 当前页对象数量
```

接近变成：

```txt
O(k) k = 本轮真正变化的对象数量
```

例如：

```txt
500 个对象，改 1 个 -> 只处理 1 个 patch
```

---

## 4. 图层面板虚拟列表

### 问题

500 个对象对应 500 个图层，如果全部渲染 DOM，会拖慢 React。

### 优化

只渲染可视区域附近的图层行：

```txt
visible rows + overscan
```

### 收益

```txt
500 行 DOM -> 约 20~30 行 DOM
减少 React render 和 DOM 节点数量
```

---

## 5. 图层行细粒度订阅

### 问题

之前 `LeftPanel` 订阅整个 `widgets`：

```ts
const widgets = useEditorStore((s) => s.widgets);
```

任意 widget 更新都会导致整个左侧面板重渲染，并重新计算：

```ts
rootIds.map((id) => widgets[id]);
```

### 优化

拆成：

```txt
LeftPanel
  -> 只负责 Tab 和组件库

LayerPanel
  -> 只订阅 rootIds

LayerRow
  -> 只订阅自己的 widgetId 对应的 widget
```

### 收益

```txt
单个 widget 更新
只影响对应图层行
不影响整个左侧面板
```

---

## 6. 图层行轻量化

### 问题

每一行常驻多个 Antd 组件：

```txt
Tooltip + Button * 5
```

即使只有 20 行，也会产生大量 React 组件。

### 优化

```txt
默认只展示名称和状态图标
操作按钮只在 hover / selected 时展示
去掉常驻 Tooltip
```

### 收益

```txt
减少 beginWork 时间
减少 Antd 组件渲染成本
```

---

## 7. 右侧属性面板本地缓冲

### 问题

之前右侧每次输入都会立刻触发完整数据流：

```txt
onChange
  -> updateWidget
  -> history.dispatch
  -> store update
  -> Canvas patch
  -> requestRenderAll
  -> RightPanel re-render
```

Slider / ColorPicker / InputNumber 会产生高频更新，容易掉帧。

### 优化

输入过程只更新本地 draft：

```txt
onChange -> setDraft
```

提交时才写 store：

```txt
InputNumber: blur / Enter
Input: blur / Enter
TextArea: blur
Slider: onChangeComplete
ColorPicker: onChangeComplete
```

### 收益

```txt
输入过程不触发全局 store 和 Canvas 重绘
减少右侧表单更新导致的掉帧
```

---

## 8. Fabric 拖拽为什么不会死循环

### 流程

```txt
用户拖动 Fabric 对象
  -> Fabric 自己实时渲染
  -> 拖动结束 object:modified
  -> updateWidget 写入 store
  -> Canvas 消费 patch
  -> patch 回 Fabric
```

### 为什么不会死循环

```txt
obj.set / setCoords 是程序式更新
不会再次触发 object:modified
```

所以不会出现：

```txt
Fabric modified -> store -> Fabric set -> modified -> store ...
```

### 后续可优化

可以给 patch 加来源：

```ts
source: 'fabric';
```

如果 patch 来自 Fabric 交互，Canvas 可以跳过重复 patch。

## 9. requestAnimationFrame 合并更新

### 适用场景

收益大的场景：

```txt
拖拽中实时同步 store
slider 实时预览
批量移动多个对象
协同编辑连续 patch
快捷键长按移动对象
```

### 思路

把一帧内多次 patch 合并：

```txt
多次 updateWidget
  -> 收集 patch
  -> requestAnimationFrame
  -> 统一 applyWidgetPatch
  -> requestRenderAll 一次
```

### 收益

```txt
减少一帧内重复 Canvas patch 和重复 render
```

### 当前是否必须

如果当前大多数操作已经是：

```txt
blur
changeComplete
object:modified
```

这种低频提交，那么收益不大，可以后续再做。

## 面试总结话术

我在这个 Fabric 编辑器里主要做的是数据更新路径优化。

最开始的问题是，几百个对象时，任意一个 widget 更新都会触发整页同步，包括属性 diff、层级检查、Canvas 渲染和 React 面板重渲染。

我把更新拆成了几类：

```txt
结构变化
属性变化
右侧表单输入
图层列表渲染
```

结构变化只处理新增、删除和层级顺序；属性变化通过 `widgetPatches` 增量同步到 Fabric；图层面板用虚拟列表和行级订阅减少 React 重渲染；右侧属性面板使用本地 draft，提交时才写 store。

这样可以避免一个小的属性变化触发整页 Canvas 和整棵 React UI 更新。

核心目标是：

```txt
让变化影响范围尽可能小
让高频输入不要直接触发全局更新
让 Canvas 渲染尽可能合并
```

## 后续还可以继续优化

### 1. source-based patch

给 patch 标记来源：

```ts
source: 'fabric' | 'panel' | 'history' | 'remote';
```

Fabric 拖拽产生的 patch 可以跳过重复回写。

### 2. requestAnimationFrame 批处理

适合实时拖拽、slider 实时预览、协同编辑。

### 3. 生产环境性能测试

Dev 模式会放大 React 和 DevTools 开销，应使用：

```bash
pnpm build
pnpm preview
```

再测 Performance。

### 4. Fabric 对象复杂度优化

减少：

```txt
shadow
filter
复杂 path
大量 Textbox
```

必要时开启：

```txt
objectCaching
```

### 5. 分层 Canvas

把画布拆成：

```txt
背景层
静态对象层
交互对象层
overlay 层
```

减少交互时的重绘范围。

### 6. 视口裁剪

大画布时只激活可视区域附近对象。

### 7. 历史命令合并

拖动、缩放、连续输入这类操作合并成一条 history command。

### 8. 更细粒度 selector

继续减少无关 store 更新导致的组件重渲染。
